# 기획 문서

이 디렉터리는 핵심 제품과 기능의 의도를 담는 살아있는 기획 문서를 저장한다.
이 문서는 시스템이 변화하는 동안 제품의 요구사항, 정책, reasoning을 계속 동기화하는 데 사용한다.

## 역할

- 기획 문서는 제품이 무엇을 의미하거나 보장해야 하는지, 그리고 왜 그것이 중요한지를 설명한다.
- 기획 문서는 구현 전에 작성해서 delivery의 입력으로 사용할 수 있다.
- 코드가 먼저 존재한 뒤에도 구현이 만든 동작, 요구사항, 정책, 용어, 범위를 기록하기 위해 기획 문서를 갱신할 수 있다.
- 기획 문서는 시간이 지나도 핵심 제품 동작과 동기화되어 있는 것이 좋다.
- 기획 문서는 세부 구현 방식을 정의하지 않는 것이 좋다.
- 구현 세부사항의 source of truth는 코드다.
- 구체적인 동작 검증은 static analysis, 테스트, type check, generator, CI가 맡는다.

## 언어 쌍

- 기획 문서는 영어와 한글 쌍 문서로 관리한다.
- 영어 문서는 `plans/en/` 아래에 둔다.
- 한글 문서는 `plans/ko/` 아래에 둔다.
- 쌍 문서는 같은 상대 경로, 파일명, 제목 계층, 주요 섹션 순서를 반드시 유지해야 한다.
- 쌍 문서가 충돌하면 의도한 정책이나 요구사항을 선택하고 같은 변경 단위에서 양쪽 문서를 모두 수정한다.

## 작성 지침

- 무엇이 필요한지와 왜 중요한지에 집중한다.
- 단계별 동작보다 오래 유지될 의도, 요구사항, 정책, 용어, 범위를 선호한다.
- 모든 문서를 같은 섹션에 억지로 맞추지 말고 기획 주제에 맞는 구조를 사용한다.
- 구현 참조는 기획 문서를 안정적인 코드나 검증과 연결하는 데 도움이 될 때만 포함한다.
- 불확실성은 모호한 문장에 숨기지 말고 명시한다.

## 문서 생성

- 해당 언어 디렉터리의 `_template.md`를 복사한다.
- 두 언어 버전을 같은 변경 단위에서 생성한다.
- 템플릿은 기획 주제에 맞게 자유롭게 조정한다.
- 제품 의도의 source of truth가 바뀌면 그 변화가 기획, 구현, 리뷰 중 어디에서 시작되었든 기획 문서를 갱신한다.
