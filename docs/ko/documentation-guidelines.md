# 문서 작성 가이드라인

오래 유지되는 프로젝트 Markdown 문서가 컨벤션, 프로젝트 동작, 장기적인 프로젝트 이해를 정의한다면 영어 문서와 한글 문서를 쌍으로 관리한다.
쌍 문서는 같은 정책을 설명해야 한다.

## 동기화 정책

- 영어 문서와 한글 문서는 쌍을 이루는 정책 문서다.
- 쌍 문서가 충돌할 때 특정 언어가 자동으로 우선하지 않는다.
- 쌍 문서가 충돌하면 영어와 한글 중 의도한 정책을 선택하고 같은 변경 단위에서 양쪽 문서를 모두 수정해야 한다.
- 충돌을 해결한 뒤 한쪽 언어를 낡은 상태로 남겨 두면 안 된다.

## 적용 범위와 예외

- 이 가이드라인은 루트 문서, 저장소 전역 문서, 앱 문서, 유지보수되는 도구 README를 포함한 오래 유지되는 프로젝트 Markdown 문서에 적용한다.
- `AGENTS.md` 같은 agent instruction file은 사람이 읽는 문서가 아니라 실행 지시문이므로 언어 쌍을 요구하지 않는다.
- `.codex/temp/` 아래 파일처럼 임시 또는 숨김 작업 노트는 제외한다.
- 생성 문서, 짧은 local note, 범위가 좁은 tool-specific file은 쌍 문서가 프로젝트 이해를 개선하지 않고 유지보수 비용만 늘린다면 단일 언어로 둘 수 있다.

## 문서의 역할

- 이 프로젝트는 문서를 engineering harness의 일부로 다룬다.
- 문서는 feedforward guidance를 제공한다. 구현 전에 의도, 경계, mental model, tradeoff, 리뷰 기준을 설명해서 판단을 형성해야 한다.
- static analysis, 테스트, type check, generator, CI는 feedback control이자 verification gate다. 구현 중이거나 구현 후에 구체적이고 정형적이며 반복 가능한 요구사항을 검증한다.
- 문서는 자동화가 context를 잘 표현하지 못하는 영역에서 판단을 안내해야 한다.
- feedback control이나 verification gate가 이미 보장하는 긴 기계적 규칙 목록을 문서가 반복해서는 안 된다.
- 기계적 요구사항이 중요하다면 문서는 강제되는 모든 세부사항을 다시 적기보다, 그 요구사항이 왜 존재하는지, 언제 중요한지, 어디에서 검증되는지를 설명해야 한다.
- 자동 검증이 도움을 주기 전에 사람이나 agent가 판단해야 하는 규칙이거나, 위반 시 정책, 정확성, 유지보수 위험이 생기는 규칙에만 영어 문서에서 `MUST`를 사용한다.
- context에 따라 달라지는 구현 선택지는, 나중에 자동 검사나 테스트가 정확한 형태를 잡아낼 수 있다면 유연한 지침을 선호한다.

## 언어 쌍

- 언어별 문서는 언어 기준 디렉터리 이름을 사용한다.
- 영어 문서는 `en`, 한글 문서는 `ko`를 사용한다.
- `docs/en/` 아래 오래 유지되는 컨벤션 문서는 같은 상대 경로와 파일명을 가진 한글 문서를 `docs/ko/` 아래에 반드시 가져야 한다.
- `apps/*/docs/en/` 아래 오래 유지되는 앱 컨벤션 문서는 같은 상대 경로와 파일명을 가진 한글 문서를 `apps/*/docs/ko/` 아래에 반드시 가져야 한다.
- 언어 디렉터리 밖의 공개 Markdown 파일이 오래 유지되는 사용자 또는 maintainer 대상 문서라면 영어를 기본 파일로 두고 한글은 `*.ko.md`로 두는 것을 권장한다. 예: `README.md`와 `README.ko.md`.

## 동기화

- 영어 문서와 한글 문서는 같은 제목 계층과 주요 섹션 순서를 반드시 유지해야 한다.
- 제목 문구는 번역할 수 있다. 계층과 순서는 맞춰야 하지만, 제목 단어를 정확히 일치시킬 필요는 없다.
- 문장별 직역은 필수가 아니다.
- 규칙, 예외, 명령어, 경로, 코드 예시, API 이름, type 이름은 양쪽 문서에서 같은 의미를 가져야 한다.
- 한 언어 문서를 변경하면 같은 PR 또는 변경 단위에서 쌍을 이루는 다른 언어 문서도 함께 변경해야 한다.

## 메타데이터

- 오래 유지되는 컨벤션 문서는 제목, 언어, audience 또는 scope, 쌍 문서를 식별하는 YAML frontmatter나 그에 준하는 metadata를 포함하는 것을 권장한다.
- 같은 문서군 안에서는 metadata key를 일관되게 유지하되, reader, tool, maintenance workflow가 사용하지 않는 key는 추가하지 않는다.
- `read_when`은 문서가 index에서 라우팅되고 그 trigger가 문서 자체에도 유용할 때만 포함한다.
- `related`는 함께 읽어야 하는 문서가 명확할 때만 포함한다.
- 문서 간 metadata link에는 상대 경로를 사용한다.

## 문서 라우팅

문서는 주제만이 아니라 언제 읽어야 하는지를 기준으로 구성해야 한다.
각 문서는 `index.md`에서 라우팅할 수 있는 명확한 작업 조건을 가져야 한다.

## 문서 구조

- 문서 제목에는 하나의 `#` heading만 사용한다.
- 주요 규칙 그룹에는 `##` heading을 사용한다.
- heading 깊이는 고정된 2단계 구조가 아니라 주제의 실제 구조를 표현하기 위해 사용한다.
- 관련 규칙, rationale, 예외, 예시, review check를 하나의 판단 영역 아래에 모을 수 있다면 `###`와 더 깊은 heading을 사용한다.
- 범위와 동기화 정책 규칙은 작업별 규칙보다 앞에 둔다.
- 기본 규칙은 예외보다 앞에 둔다.
- 관련 규칙은 하나의 긴 혼합 목록이나 반복되는 동등한 섹션으로 두지 말고 명시적인 heading 아래에 묶는다.
- 새 지침이 속한 기존 섹션이 있다면 그 섹션에 추가한다.
- rationale, 예외, review heuristic은 그것이 보완하는 규칙 가까이에 둔다.
- artifact 종류, 시간 순서, 규칙을 발견한 순서보다 독자가 내려야 하는 판단을 기준으로 묶는 것을 선호한다.
- index와 routing 문서는 얕게 유지한다. 세부 정책을 반복하지 말고 상세 문서로 라우팅한다.
- 서로 관련 없는 나머지 규칙을 담기 위한 섹션은 만들지 않는다.

## 규칙 품질

- 재사용 가능한 원칙, 장기 컨벤션, 리뷰 기준, 유지보수 이유를 표현할 때만 문서 규칙을 추가한다.
- 자동 검사가 실행되기 전에 미래의 판단을 개선하는 규칙을 추가한다.
- 규칙을 추가하기 전에 같은 판단을 이미 소유한 기존 섹션이 있는지 확인하고, 정책을 다른 곳에 반복하지 말고 그 섹션을 갱신한다.
- 임시 저장소 상태, 일회성 migration 공백, 현재의 특수한 상황만 보완하기 위한 규칙 추가는 피한다.
- 문서가 의도, 경계, 라우팅을 추가로 설명하지 않는다면 자동 검사를 그대로 반복하기 위한 정확한 syntax, formatting, structure 문서화는 피한다.
- 임시 대응은 컨벤션 문서로 승격하지 말고 작업 노트, PR 설명, 또는 해당 변경 context에 둔다.
- 장기적으로 유지할 목표 동작을 직접 쓴다. 이전 pattern이 현재 변경에만 중요하다면 `이전 X pattern을 유지하지 말고 Y를 사용한다`보다 `Y를 사용한다`를 선호한다.
- 현재 또는 과거 구현 detail은 오래 유지되는 경계, tradeoff, migration rule을 설명할 때만 언급한다.

## 작성 스타일

### 영어 문서

- 영어 agent-facing 문서는 간결하고 지시적인 문체로 작성한다.
- 간결한 규칙을 선호하되, 판단, tradeoff, 예외가 중요할 때는 rationale을 함께 포함한다.
- 정적 검사로 보장되는 규칙은 검사 내용만 반복하지 말고, 규칙의 의도를 설명한다.

### 한글 문서

- 한글 문서에서는 본문과 섹션 제목을 번역한다.
- 한글 human-facing 문서는 자연스럽게 작성하되, 요구사항의 강도는 동일하게 유지한다.
- 요구사항 강도 표현은 자연스러운 한국어 문장으로 번역한다.

### 공통 규칙

- 코드, 명령어, 파일 경로, URL, frontmatter key, API 이름, type 이름, package 이름, identifier, product name은 번역하지 않는다.
- 요구사항 표현은 정확하게 작성한다. 한쪽 언어가 다른 쪽보다 더 엄격하거나 느슨해지면 안 된다.
- 문서는 역량 있는 구현자를 위한 지침으로 작성한다. 직접적으로 쓰되, 모든 선호를 금지 규칙으로 바꾸지 않는다.
- 규칙에 tradeoff, 알려진 예외, 구현 context 의존성이 있다면 뉘앙스를 함께 설명한다.

## 요구사항 표현

- 영어 문서의 `MUST`는 반드시 지켜야 하고 일관되게 리뷰할 수 있는 규칙에 사용한다.
- 영어 문서의 `MUST NOT`은 금지된 동작에 사용한다.
- 영어 문서의 `SHOULD`는 예외가 가능하지만 기본적으로 지켜야 하는 규칙에 사용한다.
- 영어 문서의 `SHOULD NOT`은 이유가 없다면 피해야 하는 동작에 사용한다.
- 영어 문서의 `MAY`는 명시적으로 허용되는 동작에 사용한다.
- 영어 문서의 `Prefer`는 유효한 선택지 사이의 우선순위를 나타낼 때 사용한다.
- 영어 문서의 `Avoid`는 금지는 아니지만 권장하지 않는 패턴에 사용한다.
- 한글 문서에서는 requirement keyword를 그대로 남기지 말고 자연스러운 한국어 문장으로 번역한다.
- 영어 문서에서 `MUST`를 남용하지 않는다. 정책 위반, 정확성 위험, 일관되게 판단 가능한 규칙에만 사용한다.
