# E2E 테스트 컨벤션

E2E 테스트는 실행 중인 시스템 대상으로 실제 브라우저를 통해 완전한 사용자 여정을 검증한다.
여정은 사용자 또는 외부 액터가 어떤 행위를 취하는 시점에 시작되고, 사용자가 브라우저에서 결과를 확인하는 시점에 끝난다.
E2E 테스트 케이스를 작성하거나 리뷰하거나 범위를 결정할 때 이 문서를 사용한다.

## 적용 범위

E2E 테스트는 실제 브라우저를 시작하고, 실행 중인 UI를 로드하고, 사용자가 화면에서 관찰할 수 있는 것을 assertion한다.
내부 framework 상태를 검사하거나, 네트워크 호출을 mock하거나, 사용자에게 보이지 않는 값을 assertion해서는 안 된다.

## E2E가 고유하게 소유하는 것

### 완전한 사용자 여정

E2E 테스트는 실제 사용자가 제품을 경험하는 방식과 일치하는, 여러 단계와 페이지에 걸친 시나리오를 검증한다.
각 테스트는 의미 있는 사용자 목표를 나타내야 한다. 예를 들어 사용자가 노트를 업로드하고 소스 목록에 나타나는지 확인하거나, 포스트를 발행하고 포스트 페이지에 반영되는지 확인하는 식이다.
각 레이어는 시스템의 한 부분만 고립해서 실행하기 때문에, 어떤 하위 테스트 레이어도 이 경로를 커버할 수 없다.

### 렌더링

렌더링은 모든 여정이 의존하는 속성이다. API가 반환한 데이터가 브라우저에 보여야 한다.
API 응답 → React Query 캐시 → 컴포넌트 렌더 → 화면에 보이는 DOM까지의 경로는 실제 브라우저 세션에서만 실행된다.
하위 레이어 테스트는 이 경로에 미치지 못한다. UI 단위 테스트는 네트워크를 mock하고, API client 통합 테스트는 브라우저 없이 Node 환경에서 실행된다.

### 네비게이션

네비게이션은 모든 다중 단계 여정이 의존하는 속성이다. 실제 history stack을 가진 실제 브라우저 세션에서 페이지 전환, 링크 클릭, 뒤로가기, URL 파라미터 추출이 동작해야 한다.

### 시스템 구성

실행 중인 E2E 세션은 UI, API, 데이터베이스 등 모든 서비스가 동시에 연결된 상태에서 실행되는 유일한 테스트다.
환경변수 누락이나 CORS 설정 오류처럼 단일 서비스 테스트에서는 드러나지 않는 설정 오류가 E2E 세션에서는 즉시 실패로 나타난다.

### 에러 표시

E2E 테스트는 시스템 장애가 브라우저에서 사용자에게 보이는 피드백으로 나타나는지 검증한다.
에러 상태가 조용히 무시되지 않고 DOM에 실제로 렌더링되는지는 실제 브라우저 세션을 통해서만 확인할 수 있다.

## 외부 액터

일부 사용자 여정은 브라우저 바깥에서 취하는 행위에서 시작된다. 예를 들어 사용자가 Obsidian 플러그인에서 노트를 업로드하면 소스가 생성되고, 그 소스가 이후 UI에 나타난다.

이런 여정도 E2E 범위다. 사용자의 목표가 외부 행위와 브라우저에서 보이는 결과 양쪽에 걸쳐 있기 때문이다.

그러나 E2E 테스트는 외부 액터 앱 자체를 자동화해서는 안 된다.
대신, 해당 액터가 사용하는 것과 동일한 공개 API를 호출해 외부 행위를 시뮬레이션한다.
외부 액터의 올바름, 예를 들어 Obsidian 플러그인이 올바른 요청을 보내는지 여부는 해당 앱의 자체 테스트가 책임진다.

```
[E2E 테스트에서]
POST /sources        ← Obsidian 플러그인의 행위를 API 호출로 대체
      ↓
브라우저에서 UI 방문 → 소스가 목록에 보이는지 확인
```

이렇게 하면 다른 앱의 동작 책임을 E2E가 떠안지 않으면서 브라우저에서 보이는 결과에 집중할 수 있다.

## 네트워크 장애 시뮬레이션

에러 표시 속성을 테스트하려면 브라우저가 네트워크 장애를 실제로 겪어야 한다.
E2E 테스트는 API가 응답하지 않는 상황을 시뮬레이션하기 위해 API로의 네트워크 연결을 abort할 수 있다.

연결 abort는 응답을 mock하는 것과 다르다.
Mock은 실제 응답 대신 조작된 데이터를 반환하지만, abort는 브라우저가 응답 없이 연결 실패를 보게 만든다. 이는 실제 API가 다운됐을 때와 동일한 브라우저 동작이다.
이것이 E2E 테스트에서 허용되는 유일한 형태의 네트워크 인터셉션이다.
가짜 응답 데이터를 반환하기 위해 요청을 인터셉트하는 것은 허용되지 않는다.

## E2E가 소유해서는 안 되는 것

아래 항목들은 이미 하위 레이어 테스트가 커버하고 있다.
E2E 테스트는 이를 중복해서는 안 된다.

| 항목 | 소유 레이어 |
|------|------------|
| HTTP 응답 shape와 상태 코드 | API HTTP 통합 테스트 |
| API client 역직렬화와 파싱 | UI API client 통합 테스트 |
| 주어진 props로 컴포넌트 렌더링 | UI 단위 테스트 |
| 비즈니스 로직과 도메인 불변식 | API 단위 테스트 |
| 실제 어댑터 동작 (DB, 큐, 임베더) | API 어댑터 통합 테스트 |
| 외부 액터 앱의 동작 | 해당 액터 앱의 자체 테스트 |

이 항목들을 E2E 테스트에서 중복하면 테스트 비용과 불안정성만 높아지고 신뢰도는 늘지 않는다.
이 레이어의 실패가 E2E 테스트에서 드러난다면, E2E 커버리지를 확장하는 대신 해당 책임이 있는 하위 레이어의 커버리지를 추가하거나 수정하는 신호로 받아들인다.

## 결정 규칙

다음에 해당하면 E2E 테스트를 작성한다.

- 여러 단계 또는 페이지에 걸친 완전한 사용자 목표를 나타내는 테스트인 경우.
- 브라우저 안에서의 행위든 외부 액터의 행위든, 어떤 행위가 브라우저에서 보이는 결과로 이어지는지 검증하는 경우.
- 실행 중인 시스템 대상으로 실제 브라우저를 통해서만 관찰 가능한 동작인 경우.

다음에 해당하면 E2E 테스트를 작성하지 않는다.

- 하위 레이어에서 신뢰도를 크게 잃지 않고 동일한 동작을 검증할 수 있는 경우.
- 단일 레이어 또는 단일 앱 안에 포함된 로직을 검증하는 경우.
- 다른 곳에서 이미 커버하고 있는 HTTP 계약, 파싱, props 기반 렌더링, 외부 액터 동작을 중복하는 경우.

## 테스트 셋업

E2E 테스트는 데이터베이스에 직접 삽입하지 않고 공개 API를 통해 사전 조건을 설정하는 것이 좋다.
이렇게 하면 실제 사용자와 외부 통합이 사용하는 것과 동일한 프로토콜을 사용하고, 내부 persistence 세부사항에 테스트가 의존하는 것을 방지하며, 셋업 단계 자체도 검증된 시스템 경계 안에 둘 수 있다.
